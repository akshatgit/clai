# Design: `clai exec` and `clai accept` Commands

## 1. Codebase Analysis

### 1.1 CLI Framework

The project uses **Commander.js v12** (`commander` package) as the CLI framework. The single entry point is `src/index.js`, which imports `{ program }` from `commander` and registers commands via the chained builder API:

```js
program
  .command('name <required-arg> [optional-arg]')
  .description('...')
  .option('--flag <value>', 'description')
  .action((arg, opts) => { ... })
```

The file ends with `program.parse()` to kick off argument parsing.

### 1.2 Existing Commands (7 total)

| Command | Pattern | Location in `src/index.js` |
|---------|---------|---------------------------|
| `start <goal>` | Plans a DAG, optionally runs it | ~line 141 |
| `run <session-id>` | Runs pending tasks or re-runs one | ~line 193 |
| `status <session-id>` | Shows session/task status | ~line 222 |
| `list` | Lists all sessions | ~line 266 |
| `logs <session-id>` | Shows JSONL event log | ~line 282 |
| `viz <session-id>` | Terminal or HTML DAG visualization | ~line 429 |
| `containers` | Lists Docker task containers | ~line 448 |
| `serve` | HTTP server for browser UI | ~line 418 |

New commands should be registered in the `// ─── Commands ───` section of `src/index.js`, after the existing commands and before `program.parse()`.

### 1.3 Session & Task Storage

**Storage mechanism:** File-based JSON. Each session is a single JSON file.

| Artifact | Path | Format |
|----------|------|--------|
| Session state | `sessions/<sessionId>.json` | JSON |
| Event log | `logs/<sessionId>.jsonl` | Newline-delimited JSON |

**State module:** `src/state.js` exports:

| Function | Signature | Purpose |
|----------|-----------|---------|
| `createSession` | `(goal: string, dagTasks: object[]) → session` | Create + save a new session |
| `loadSession` | `(sessionId: string) → session` | Read session from disk; throws if not found |
| `saveSession` | `(session: object) → void` | Write session to disk |
| `listSessions` | `() → { id, goal, status, created_at }[]` | List all sessions |
| `resetTask` | `(session, taskId) → void` | Reset a task to pending + save |
| `generateId` | `(prefix?) → string` | Generate random ID |
| `SESSIONS_DIR` | `string` (exported let) | Path to sessions directory |
| `LOGS_DIR` | `string` (exported let) | Path to logs directory |

### 1.4 Task Schema

Each task object in `session.dag.tasks[taskId]` has:

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | e.g. `task_1` |
| `title` | `string` | Short title |
| `description` | `string` | Detailed spec |
| `dependencies` | `string[]` | IDs of prerequisite tasks |
| `complexity` | `'low' \| 'medium' \| 'high'` | Model selection hint |
| `docker_image` | `string` | Docker image for container |
| `completion_criteria` | `string[]` | Success conditions |
| `tests` | `string[]` | Shell validation commands |
| `input_paths` | `string[]` | Read paths (defaults `['.']`) |
| `output_paths` | `string[]` | Write paths |
| `status` | `'pending' \| 'running' \| 'completed' \| 'failed' \| 'skipped'` | Current state |
| `result` | `string \| null` | Task output text |
| `started_at` | `string \| null` | ISO timestamp |
| `completed_at` | `string \| null` | ISO timestamp |
| `attempts` | `number` | Execution attempt count |

### 1.5 Docker Container Naming

Container names follow the pattern: `clai-<sessionId>-<taskId>`

Generated by `containerName(sessionId, taskId)` in `src/docker.js`:

```js
export function containerName(sessionId, taskId) {
  return `clai-${sessionId}-${taskId}`
}
```

Containers are started with `docker run -d --name <name> ...` and persist after the task completes (they run `tail -f /dev/null` to stay alive). The existing `containers` command lists them via `docker ps -a --filter name=clai-`.

### 1.6 Error Handling Patterns

Existing commands follow a consistent pattern:

1. Wrap `loadSession()` in a try/catch
2. On error, call `fail(err.message)` (prints red `✗` + message)
3. Call `process.exit(1)`

```js
try {
  session = loadSession(sessionId)
} catch (err) {
  fail(err.message)
  process.exit(1)
}
```

For Docker errors, the project uses `execSync` or spawns child processes and checks exit codes.

## 2. Command Design

### 2.1 `clai exec <session-id> <task-id>`

**Purpose:** Shortcut for `docker exec -it clai-<sessionId>-<taskId> sh`

**Registration location:** `src/index.js`, in the Commands section, before `program.parse()`.

**Commander definition:**

```js
program
  .command('exec <session-id> <task-id>')
  .description('Open a shell inside a task\'s Docker container')
  .action((sessionId, taskId) => { ... })
```

**Handler function signature:**

```js
function handleExec(sessionId, taskId)
```

**Implementation steps:**

1. Load session via `loadSession(sessionId)` — validates session exists.
2. Validate that `taskId` exists in `session.dag.tasks`.
3. Build the container name via `containerName(sessionId, taskId)`.
4. Check if the container is running by executing:
   ```bash
   docker inspect -f '{{.State.Running}}' <containerName>
   ```
   - If the command fails or returns `false`, print a clear error:
     `"Container <name> is not running. Start it with: clai run <session-id> --docker"` and `process.exit(1)`.
5. Replace the current process with `docker exec -it <containerName> sh` using `execSync` with `{ stdio: 'inherit' }` (so the user gets an interactive terminal).
   - Alternatively, use `child_process.spawnSync('docker', ['exec', '-it', name, 'sh'], { stdio: 'inherit' })` which handles signals properly.

**Error cases:**
- Session not found → `fail('Session not found: <id>')` + exit 1
- Task not found in session → `fail('Task not found: <id>')` + exit 1
- Container not running → `fail('Container <name> is not running. ...')` + exit 1
- Docker not available → inherits from spawn failure

**Storage calls:**
- `loadSession(sessionId)` — read-only, to validate session/task existence
- `containerName(sessionId, taskId)` — pure function, builds the name

**No writes to session state** — this command is purely interactive.

### 2.2 `clai accept <session-id> <task-id> [--message <text>]`

**Purpose:** Manually mark a task as completed so downstream tasks can proceed. Useful when the user has manually modified files inside a container and wants to continue the session from that point.

**Registration location:** `src/index.js`, in the Commands section, before `program.parse()`.

**Commander definition:**

```js
program
  .command('accept <session-id> <task-id>')
  .description('Manually mark a task as completed (skip AI execution)')
  .option('--message <text>', 'Optional message to record as the task result')
  .action((sessionId, taskId, opts) => { ... })
```

**Handler function signature:**

```js
function handleAccept(sessionId, taskId, opts)
```

where `opts.message` is the optional `--message` value.

**Implementation steps:**

1. Load session via `loadSession(sessionId)`.
2. Validate that `taskId` exists in `session.dag.tasks`.
3. Read the task object: `const task = session.dag.tasks[taskId]`.
4. Update the following three fields:
   - **`status`**: Set to `'completed'`
   - **`result`**: Set to `opts.message ?? 'Manually accepted'` (a default string if no message given)
   - **`completed_at`**: Set to `new Date().toISOString()`
5. Save session via `saveSession(session)`.
6. Emit a hook event for the log trail:
   ```js
   emit('task:completed', {
     sessionId: session.id,
     taskId,
     taskTitle: task.title,
     duration: 0,
     resultLength: task.result.length,
   })
   ```
7. Print success message: `success(\`[${taskId}] ${task.title} — manually accepted\`)`
8. Print info about next steps: show which tasks are now ready (using `getReadyTasks`), and suggest `clai run <session-id>` to continue.

**Error cases:**
- Session not found → `fail('Session not found: <id>')` + exit 1
- Task not found in session → `fail('Task not found: <id>')` + exit 1

**Storage calls (reads and writes):**
- `loadSession(sessionId)` — read
- `saveSession(session)` — write (after updating task fields)
- `emit('task:completed', {...})` — writes to `logs/<sessionId>.jsonl`

**Fields updated on the task object:**

| Field | New Value | Why |
|-------|-----------|-----|
| `status` | `'completed'` | Marks the task as done so `getReadyTasks()` sees it as satisfied for downstream dependencies |
| `result` | `opts.message ?? 'Manually accepted'` | Records context about what was done manually; shown by `clai status --result <task-id>` |
| `completed_at` | `new Date().toISOString()` | Timestamp of acceptance; used for duration display in `clai status` |

## 3. Imports Needed

Both commands will use existing imports already present at the top of `src/index.js`:

| Import | Source | Used By |
|--------|--------|---------|
| `loadSession`, `saveSession` | `./state.js` | Both commands |
| `containerName` | `./docker.js` | `exec` command |
| `emit` | `./hooks.js` | `accept` command |
| `getReadyTasks` | `./dag.js` | `accept` command (optional, for next-steps hint) |
| `execSync` / `spawnSync` | `child_process` | `exec` command |

All of these are **already imported** in `src/index.js`. No new dependencies are required.

---

## 4. Code Placement

New commands will be inserted in `src/index.js` between the existing `containers` command block and the final `program.parse()` call. This keeps them grouped logically with other operational commands (containers, exec, accept).

Suggested ordering:

```
// ... existing commands ...

// CONTAINERS: list Docker containers for task runs
program.command('containers') ...

// EXEC: open a shell inside a task container
program.command('exec <session-id> <task-id>') ...

// ACCEPT: manually mark a task as completed
program.command('accept <session-id> <task-id>') ...

program.parse()
```

## 5. Testing Considerations

### Unit tests for `accept`:
- Load a session with a pending task, run accept, verify `status === 'completed'`, `result` contains the message, and `completed_at` is set.
- Verify downstream tasks become ready after accept.
- Verify that accepting an already-completed task still works (idempotent).

### Unit tests for `exec`:
- Mock `execSync`/`spawnSync` to verify the correct Docker command is constructed.
- Verify error when container is not running.
- Verify error when session/task doesn't exist.

### Integration:
- Both commands can be tested via `node src/index.js exec ...` and `node src/index.js accept ...` with real session files in a temp directory (using `_configure()` from state.js to redirect storage).
